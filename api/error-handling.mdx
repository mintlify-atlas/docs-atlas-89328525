title: "Error handling"

All backend services in Mi Colección de Juegos return errors in the [RFC 7807 Problem Details](https://www.rfc-editor.org/rfc/rfc7807) format. This gives every error response a consistent, machine-readable structure that both the frontend and API consumers can rely on.

<Note>
  RFC 7807 defines a standard JSON format for HTTP error responses. Using `"type": "about:blank"` indicates that the HTTP status code itself fully describes the problem category, with the `detail` field providing specific context for this occurrence.
</Note>

## Error response structure

Every error response has `Content-Type: application/problem+json` and the following fields:

<ResponseField name="type" type="string">
  A URI reference identifying the problem type. Most errors use `"about:blank"`. The `recibeJson()` library uses `"/errors/datosnojson.html"` when the request body is not valid JSON.
</ResponseField>

<ResponseField name="title" type="string">
  A short, human-readable summary of the problem.
</ResponseField>

<ResponseField name="status" type="number">
  The HTTP status code for this occurrence of the problem. Mirrors the actual HTTP response status code.
</ResponseField>

<ResponseField name="detail" type="string">
  A human-readable explanation specific to this occurrence. This field is **optional** — it is only included when additional context beyond the title is available. The frontend reads this field to display user-facing error messages.
</ResponseField>

```json example error response
{
  "type": "about:blank",
  "title": "Faltan datos requeridos: Título y Género no pueden estar vacíos.",
  "status": 400
}
```

## Error codes

| Status | When it occurs |
| ------ | -------------- |
| `400`  | A required field is missing or contains invalid data |
| `405`  | The endpoint was called with the wrong HTTP method |
| `500`  | An unexpected internal server error occurred |
| `502`  | An external service (FreeToGame API) was unreachable |

## Error classes

The error system is built on two PHP classes.

### `ProblemDetailsError`

Defined in `php/error_handling.php`. The base class for all application errors. Accepts a title string, HTTP status code, optional detail, and optional type URI.

```php php/error_handling.php
class ProblemDetailsError extends Exception {
    public function __construct(
        $title,
        $status = 400,
        $detail = null,
        $type = "about:blank"
    ) { ... }

    public function sendResponse() {
        http_response_code($this->status);
        header('Content-Type: application/problem+json');

        $response = [
            "type"   => $this->type,
            "title"  => $this->title,
            "status" => $this->status
        ];

        // detail is only included when non-null
        if ($this->detail !== null) {
            $response["detail"] = $this->detail;
        }

        echo json_encode($response);
        exit();
    }
}
```

### `ProblemDetailsException`

Defined in `php/lib/ProblemDetailsException.php`. Accepts a pre-built content array and sends it directly, providing more control over the full response payload.

```php php/lib/ProblemDetailsException.php
class ProblemDetailsException extends Exception {
    public function __construct(array $content) {
        parent::__construct($content['title'] ?? 'Error');
        $this->content = $content;
    }

    public function sendResponse() {
        $status = $this->content['status'] ?? 400;
        http_response_code($status);
        header("Content-Type: application/problem+json; charset=utf-8");
        echo json_encode($this->content);
        exit();
    }
}
```

## Global exception handler

`php/error_handling.php` registers a global exception handler using `set_exception_handler`. It catches any uncaught `ProblemDetailsError` or `ProblemDetailsException` and calls `sendResponse()`. All other exceptions are caught and returned as a generic 500 response.

```php php/error_handling.php
function muestraError(Throwable $e) {
    if ($e instanceof ProblemDetailsError || $e instanceof ProblemDetailsException) {
        $e->sendResponse();
    } else {
        http_response_code(500);
        header('Content-Type: application/problem+json');
        echo json_encode([
            "type"   => "about:blank",
            "title"  => "Error interno del servidor",
            "status" => 500,
            "detail" => $e->getMessage()
        ]);
        exit();
    }
}

set_exception_handler('muestraError');
```

## Frontend error handling

The frontend uses `js/lib/consume.js` and `js/lib/ProblemDetailsError.js` to detect and parse Problem Details responses. When a `fetch` call returns a non-2xx response with `Content-Type: application/problem+json`, `consume()` throws a `ProblemDetailsError` containing the parsed body.

```javascript js/lib/consume.js
export async function consume(servicio) {
    const respuesta = await servicio;
    if (respuesta.ok) {
        return respuesta;
    } else {
        const contentType = respuesta.headers.get("Content-Type");
        if (contentType !== null && contentType.startsWith("application/problem+json"))
            throw new ProblemDetailsError(await respuesta.json());
        else
            throw new Error(respuesta.statusText);
    }
}
```

`js/lib/muestraError.js` then reads the `title` and `detail` fields from the thrown error and displays them to the user via `alert()` and `console.error()`.
